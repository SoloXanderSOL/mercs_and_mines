## 1. PHASE 0: HACKATHON MVP SCOPE

Your immediate task is to build a working prototype that proves the core loop and the Solana
integration. You will build exactly this and nothing more:

* **TEEPIN Wallet Connection:** Implement the Solana Seeker TEEPIN authentication flow. The
  player authenticates with their TEEPIN once at session start, and the Seed Vault issues a
  session authorization covering all micro-transactions for the next 2 hours. There must be
  zero Phantom Wallet pop-ups during normal play.

* **The Trust 'Founding Courtesy':** On account creation, trigger the first on-chain
  transaction: The Trust delivers 500 He3 to the player's starter Cryo-Vat. This is a
  one-way delivery (Trust → Player) — the player exchanges nothing in the demo. Trust
  Standing initializes at 0 (Neutral). This is the on-chain proof-of-concept for the
  Solana integration pipeline.

  NOTE — DEMO SIMPLIFICATION: In the production game, this sequence is a full scripted
  Story Event at tutorial Beat 3.6, triggered when the player builds Cryo-Vats, and
  involves an Iron Ore exchange. The account-creation trigger used here is a placeholder
  for the hackathon only. Do not architect this as a permanent onboarding flow.

* **One AP/AT Combat Resolution:** A single mock Sector instance where the player resolves
  one combat engagement via the AP vs. AT tick engine. The Live Tactical Dashboard text feed
  must scroll in real time as the server processes ticks. The After-Action Report must be
  generated by replaying the input log — not from a stored state snapshot.

## 2. NON-NEGOTIABLE ARCHITECTURE (THE DETERMINISM RULE)

This is a high-stakes Web3 game where the input log acts as a financial audit trail. The
server must be perfectly deterministic. When writing backend game logic, you must obey the
following rules:

* **Authoritative Server / Thin Client:** The server is the only source of truth. The client
  renders output and sends input — nothing more. You must not implement client prediction.

* **Single Source of Randomness:** All game randomness must come from a single, explicit
  PRNG instance initialized with a known seed. You must NEVER use `Math.random()`, `rand()`,
  or platform RNGs inside the simulation.

* **No Wall-Clock Time:** Simulation logic must never call `now()` or rely on real-time
  durations inside `Step()`. Derive "time" strictly from the tick count.

* **Append-Only Input Logs:** Log player inputs, timer expiries, and server decisions. Do
  NOT log full game states. The log must be immutable and strictly ordered. Each entry
  includes tick index, sequence number, and build version.

## 3. NETWORKING AND DATA DEFAULTS

* **Protocols:** Use HTTP/HTTPS + JSON for all meta-actions (login, inventory). Use
  WebSockets for the Live Tactical Dashboard tick feed and real-time combat logs.
  Do NOT use UDP.

* **Database:** Use a Relational DB (e.g., PostgreSQL) for accounts, player profiles, and
  the $ASH ledger. Use a Key-Value store (e.g., Redis) for live Sector game state and active
  deployment timers. Use an append-only log store for input logs — treat log integrity as a
  financial requirement, not a DB convenience.

## Frontend (`app/`)
HTML5 / Vanilla JS / CSS only. No build step, no TypeScript, no npm.
`@solana/web3.js` loaded via CDN. Axum serves `app/` as static files via `tower-http::ServeDir`.
The Rust mandate does not apply to `app/`.
